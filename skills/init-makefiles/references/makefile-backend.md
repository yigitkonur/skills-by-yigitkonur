# Backend (Railway) — `railway.toml` baseline, deploy targets, multi-service rules

This file is the canonical source for the Railway-deploy section of any Makefile generated by this skill. It applies to **Scenario C** (frontend + custom backend) and **Scenario E** (multi-service Railway). Railway is the only backend provider this skill covers — for ad-hoc Railway CLI ops outside the deploy flow (logs, scale, restart, env editing), the `run-railway` skill is the cross-skill handoff. Combine this file with `makefile-base.md` (preamble + helpers) and `env-vars-conventions.md` (where Railway env vars live per scenario).

All facts below verified May 2026 against `https://docs.railway.com`.

## What Railway is in 2026

Railway is a Platform-as-a-Service for **long-running containers** — not a serverless / FaaS platform. Each service is a process that boots, registers a healthcheck, then takes traffic. Restart policy and healthcheck behavior are first-class config, not add-ons. Reference: `https://docs.railway.com/reference/services`.

Two consequences for the Makefile:

1. The deploy is `railway up` against a service that already exists in the project — Railway does not `vercel deploy`-style spawn ephemeral envs. Each service has identity (name, env, domain).
2. `PORT` is supplied by Railway at runtime; the app **must** bind to `0.0.0.0:$PORT`. Binding `localhost:3000` is the #1 cause of "Application Failed to Respond" 502s.

## The minimal `railway.toml` baseline

Every Railway service the skill scaffolds gets a `railway.toml` at the service's root path. The baseline (verbatim — drop-in for HTTP services):

```toml
[build]
builder = "RAILPACK"

[deploy]
startCommand = "node dist/server.js"
healthcheckPath = "/health"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 5
```

Why each line is non-optional:

| Field | Why |
|---|---|
| `builder = "RAILPACK"` | 2026 default builder. Railpack replaced Nixpacks as the default in 2025 (Nixpacks is legacy). Auto-detects Node, Python, Go, etc. |
| `startCommand` | Single source of truth for how the app boots. Even when a `Dockerfile` is used, Railway honors this OVER the Dockerfile's `ENTRYPOINT`/`CMD`. |
| `healthcheckPath = "/health"` | Mandatory in practice. Without it, Railway routes traffic to a half-booted app. |
| `healthcheckTimeout = 300` | 5 minutes for slow boots (Prisma generate, JIT warmup). Tune down for fast boots. |
| `restartPolicyType = "ON_FAILURE"` | Restart only on non-zero exit; no restart loops on healthy stops. |
| `restartPolicyMaxRetries = 5` | Bounds restart loops — prevents 100% CPU on a permanently-broken deploy. |

Source: `https://docs.railway.com/config-as-code/reference` (2026-05-08).

### Dockerfile variant

When a `Dockerfile` exists at the service root, OR when explicit Docker is desired (multi-step migrations, native packages, identical local-vs-prod builds):

```toml
[build]
builder = "DOCKERFILE"
dockerfilePath = "Dockerfile"
watchPatterns = ["src/**", "Dockerfile", "package.json", "package-lock.json"]

[deploy]
startCommand = "node dist/server.js"   # overrides Dockerfile ENTRYPOINT/CMD
healthcheckPath = "/health"
healthcheckTimeout = 300
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 5
```

Notes:

- `dockerfilePath` is relative to the service's root, not the repo root.
- `watchPatterns` controls which file changes trigger a rebuild on push deploys; not used in `railway up --ci` (CI deploys always rebuild).
- `deploy.startCommand` overrides the Dockerfile `CMD` and `ENTRYPOINT`. To honor the Dockerfile's `CMD`, omit `startCommand` entirely. Keep `startCommand` if any boot logic differs between dev and prod.

### Worker variant (no HTTP listener)

For BullMQ consumers, queue runners, sweepers — anything that does NOT serve HTTP:

```toml
[build]
builder = "RAILPACK"

[deploy]
startCommand = "node dist/worker.js"
restartPolicyType = "ALWAYS"
restartPolicyMaxRetries = 10
```

Differences vs HTTP variant:

- **Omit `healthcheckPath`** — there is no HTTP server to probe. Setting one would cause Railway to mark the service unhealthy and restart in a loop.
- **`restartPolicyType = "ALWAYS"`** — workers should restart even on clean exit (graceful shutdown after queue drain still wants to come back up).
- **Higher `maxRetries`** — workers can be transient-failure-prone (Redis flap, DB connection drop) and should retry more aggressively than a stateless web service.

### Per-environment overrides

`railway.toml` supports per-environment override blocks. The most common: harden production, leave dev permissive.

```toml
[build]
builder = "RAILPACK"

[deploy]
startCommand = "node dist/server.js"
healthcheckPath = "/health"
restartPolicyType = "ON_FAILURE"

[environments.production.deploy]
healthcheckTimeout = 600
restartPolicyMaxRetries = 10

[environments.pr]
# PR Environments — preview-style ephemeral envs per pull request
[environments.pr.deploy]
healthcheckTimeout = 60
restartPolicyMaxRetries = 1
```

Reference: `https://docs.railway.com/config-as-code/reference#environments` (2026-05-08).

Code values **silently override** dashboard values — dashboard does NOT reflect what's in the file. The agent commenting this in the file header is a kindness:

```toml
# Generated by init-makefiles. Code values override dashboard values silently.
# Edit here, not in Railway UI.
```

## Single-service `make deploy-railway`

Scenario C (frontend + backend) ships exactly one Railway service. The deploy target:

```makefile
deploy-railway: _check-railway-tokens
	@printf "$(D)→ railway up --ci -s $(RAILWAY_SERVICE)$(Z)\n"
	@railway up --ci -s $(RAILWAY_SERVICE)
	@$(MAKE) verify
```

The `--ci` flag disables the interactive log stream so the recipe exits when the upload is acknowledged, not when the build finishes. The `-s <service>` is **never** optional — even single-service repos benefit from being explicit because linked-service local state (`.railway/`) is not portable across machines or CI.

The preflight helper:

```makefile
_check-railway-tokens:
	@if [ -z "$${RAILWAY_TOKEN:-}" ]; then \
	  printf "$(R)RAILWAY_TOKEN not set$(Z)\n"; \
	  printf "$(D)Create a project token: Project → Settings → Tokens$(Z)\n"; \
	  printf "$(D)https://docs.railway.com/guides/cli#authenticating-with-cli$(Z)\n"; \
	  exit 1; \
	fi
	@command -v railway >/dev/null 2>&1 || { \
	  printf "$(R)railway CLI not found$(Z)  $(D)bash <(curl -fsSL railway.com/install.sh)$(Z)\n"; exit 1; }
```

Default knobs the Makefile declares near the preamble:

```makefile
RAILWAY_SERVICE ?= api          # the service name Railway dashboard shows
```

## Multi-service Scenario E: `deploy-all`

When the project has multiple Railway services (`apps/api`, `apps/worker`, `apps/cron`), the agent generates a `deploy-all` target that fires the uploads in parallel.

```makefile
deploy-all: _check-railway-tokens
	@printf "$(D)→ deploying api, worker, cron in parallel$(Z)\n"
	@( railway up --ci -s api & ) ; ( railway up --ci -s worker & ) ; ( railway up --ci -s cron & ) ; wait
	@$(MAKE) verify
```

Each `( ... & )` subshell launches a `railway up` in the background. `wait` blocks until all three finish (or fail). The recipe inherits `pipefail` from `.SHELLFLAGS`, so any non-zero exit from a service's upload propagates to the recipe.

Per-service deploys also exist for the case where the user wants to ship one service without touching the others:

```makefile
deploy-api: _check-railway-tokens
	@railway up --ci -s api

deploy-worker: _check-railway-tokens
	@railway up --ci -s worker

deploy-cron: _check-railway-tokens
	@railway up --ci -s cron
```

`deploy-all` is the user-facing target; the per-service targets are the surgical option.

### First-deploy serial bootstrap warning

Railway services can reference each other's variables: `${{ api.RAILWAY_PUBLIC_DOMAIN }}` evaluates to `api.up.railway.app` once `api` has a public domain. **On first deploy**, the referenced service must have a domain before the referrer can resolve the variable. Parallel deploy on first setup will race: `worker` may try to read `${{ api.RAILWAY_PUBLIC_DOMAIN }}` before `api` exists.

The agent detects this by grepping each `railway.toml` for `${{ ... }}` patterns:

```bash
# At generation time:
ref_count=$(grep -rEh '\$\{\{[[:space:]]*[A-Za-z][A-Za-z0-9_-]*\.' apps/*/railway.toml 2>/dev/null | wc -l)
if [ "$ref_count" -gt 0 ]; then
  echo "→ first-deploy must be serial (reference variables detected)"
fi
```

When references are detected, the generated Makefile emits a one-time hint above `deploy-all`:

```makefile
# First-deploy hint:
#   Reference variables detected (e.g. ${{ api.RAILWAY_PUBLIC_DOMAIN }}).
#   On the first deploy, run services in dependency order:
#     make deploy-api && make deploy-worker && make deploy-cron
#   Once all services have stable domains, `make deploy-all` is safe to run in parallel.
```

The agent does NOT auto-rewrite `deploy-all` — the user owns the dependency graph. The hint is sufficient.

### Parallel safety rules

| Situation | Parallel safe? |
|---|---|
| All services already exist, no migrations, no first-time references | YES — `deploy-all` is safe |
| Reference variables on first deploy (e.g. `${{ api.RAILWAY_PUBLIC_DOMAIN }}`) | NO — referenced service must deploy first |
| Two services run a DB migration in `preDeployCommand` | NO — race; one service must own migrations |
| One service runs migrations, others don't | YES — but gate `deploy-all` on the migrating service finishing first |
| Worker depends on a domain provided by `api` (in code, not in env) | YES if env-resolved; NO if hardcoded — fix the hardcode |

Migrations are **always serial**. Pick one service (typically `api`) to own `prisma migrate deploy` (or equivalent). The others read the same DB but never run migrations. Multiple services racing on the same migration table is a corruption hazard.

A migration target the agent can generate when migrations are part of the contract:

```makefile
migrate: _check-railway-tokens
	@printf "$(D)→ running migrations from api service$(Z)\n"
	@railway run --service api -- npm run migrate:deploy
```

## `RAILWAY_TOKEN` vs `RAILWAY_API_TOKEN`

Railway has two token kinds. The Makefile uses **only the project token**, never the account token.

| Token | Scope | Use case | Where set |
|---|---|---|---|
| `RAILWAY_TOKEN` | Project-scoped (this project's services only) | CI/CD, `make deploy-railway`, GitHub Actions | Dashboard → Project → Settings → Tokens |
| `RAILWAY_API_TOKEN` | Account-scoped (every project on this account) | Account-level scripts, `railway whoami` from CLI | Dashboard → Account → Tokens |

Why this matters: a leaked `RAILWAY_API_TOKEN` is catastrophic — the holder can read/write/delete *every project* on the account. A leaked `RAILWAY_TOKEN` is bounded to one project, so blast radius is small.

The Makefile preflight (`_check-railway-tokens`) refuses to deploy if `RAILWAY_TOKEN` is missing, and prints the dashboard URL to create one. It does **not** fall back to `RAILWAY_API_TOKEN` — that would silently broaden scope.

In CI, the `gh secret set RAILWAY_TOKEN ...` step from `ci-cd-workflow.md` accepts only the project token. The skill never asks for `RAILWAY_API_TOKEN`.

## Reference variables — `${{ ... }}` syntax

Railway's per-service env supports cross-service and shared references. Resolved at deploy time, embedded into the runtime env.

| Pattern | Example | Resolves to |
|---|---|---|
| Cross-service | `${{ api.RAILWAY_PUBLIC_DOMAIN }}` | `api.up.railway.app` (or custom domain) |
| Shared | `${{ shared.STRIPE_KEY }}` | A project-scoped variable shared across services |
| Database addon | `${{ Postgres.DATABASE_URL }}` | The internal `postgres.railway.internal:5432/railway` URL |
| Built-in | `${{ RAILWAY_PUBLIC_DOMAIN }}` | This service's own public domain |

Built-in service variables (Railway sets these automatically — DO NOT shadow):

| Var | Value |
|---|---|
| `RAILWAY_PUBLIC_DOMAIN` | `<service>.up.railway.app` or the user's custom domain |
| `RAILWAY_PRIVATE_DOMAIN` | `<service>.railway.internal` — only resolves inside Railway's private network |
| `RAILWAY_TCP_PROXY_DOMAIN`, `RAILWAY_TCP_PROXY_PORT` | TCP proxy endpoint (when TCP proxying is enabled) |
| `PORT` | The port Railway expects the service to bind to |
| `RAILWAY_PROJECT_ID`, `RAILWAY_SERVICE_ID`, `RAILWAY_ENVIRONMENT_ID` | Identifiers for logging context |
| `RAILWAY_DEPLOYMENT_ID` | This deployment's UUID |
| `RAILWAY_GIT_COMMIT_SHA`, `RAILWAY_GIT_BRANCH` | Set when the deploy was triggered from GitHub |

Source: `https://docs.railway.com/reference/variables` (2026-05-08). See also `env-vars-conventions.md` for the cross-skill view.

## The `0.0.0.0:$PORT` bind rule

The single most common Railway 502: the app binds `localhost:3000` (or another fixed port), Railway's edge can't reach it because Railway's container networking only routes external traffic to interfaces bound on `0.0.0.0` (all interfaces). The healthcheck fails. Railway routes traffic to the previous deployment, or returns "Application Failed to Respond" if no previous deploy.

**Rule: bind `0.0.0.0:$PORT` for HTTP services. Always.**

| Stack | Right |
|---|---|
| Node + Express | `app.listen(process.env.PORT \|\| 3000, '0.0.0.0', () => { ... })` |
| Node + Hono | `serve({ fetch: app.fetch, port: Number(process.env.PORT) \|\| 3000, hostname: '0.0.0.0' })` |
| Node + Fastify | `await app.listen({ port: Number(process.env.PORT) \|\| 3000, host: '0.0.0.0' })` |
| Python + Uvicorn | `uvicorn.run("main:app", host="0.0.0.0", port=int(os.environ.get("PORT", 8000)))` |
| Python + Gunicorn | `gunicorn -b 0.0.0.0:$PORT main:app` (in `Procfile` or `startCommand`) |
| Go + net/http | `http.ListenAndServe(":" + os.Getenv("PORT"), nil)` — note `:PORT` (no host) binds all interfaces in Go |
| Go + Chi/Gin | `http.ListenAndServe("0.0.0.0:" + os.Getenv("PORT"), router)` (explicit) |
| Bun | `Bun.serve({ port: Number(process.env.PORT), hostname: '0.0.0.0', fetch })` |
| Deno | `Deno.serve({ port: Number(Deno.env.get('PORT')), hostname: '0.0.0.0' }, handler)` |

The Makefile cannot enforce this — it's app code. But the agent generating the `railway.toml` and Makefile prints a one-line warning if it can detect a `localhost` or `127.0.0.1` bind in the obvious entry-point files (`src/index.ts`, `src/server.ts`, `apps/api/src/main.ts`).

## Healthcheck rules

Railway's healthcheck is an HTTP GET against `healthcheckPath` from inside the container's network. The path must:

- Return **HTTP 200** when the service is ready to receive real traffic
- Be **fast** — Railway probes frequently; a slow `/health` adds latency to every probe
- Be **honest** — return 5xx if any critical dependency is down (DB unreachable, Redis flap)

Minimal Express healthcheck:

```ts
import { db } from './db';
app.get('/health', async (_req, res) => {
  try {
    await db.$queryRaw`SELECT 1`;
    res.status(200).json({ status: 'ok' });
  } catch {
    res.status(503).json({ status: 'db_unreachable' });
  }
});
```

Without `healthcheckPath` set, Railway considers the service healthy as soon as the process binds the port — which is too early. The deploy may "succeed" but real requests fail because the app is mid-startup. Setting `healthcheckPath` is mandatory for HTTP services.

For workers (no HTTP listener), there is no healthcheck — Railway treats process-alive as healthy. This is why workers use `restartPolicyType = "ALWAYS"`: any process exit triggers a restart, regardless of cause.

Reference: `https://docs.railway.com/reference/healthchecks` (2026-05-08).

## Common failure modes

| Symptom | Root cause | Fix |
|---|---|---|
| 502 "Application Failed to Respond" | App binds `localhost:PORT` not `0.0.0.0:$PORT` | Fix the app's bind address; redeploy |
| Deploy succeeds, requests fail | `healthcheckPath` not set; Railway routes to half-booted app | Add `healthcheckPath = "/health"` to `railway.toml`; implement endpoint |
| `railway up` errors with "no service specified" | Multi-service repo without `-s` flag | Pass `-s <service>` explicitly |
| `railway up` errors with "no project linked" | `.railway/` missing or wrong project | Project token in `RAILWAY_TOKEN` env auto-targets the project; or run `railway link` interactively once |
| Build hits OOM | Image bundling 2 GB of `node_modules` | Multi-stage Dockerfile; `--omit=optional`; `.dockerignore` |
| Reference variable resolves to literal `${{ ... }}` | First deploy of referenced service hasn't completed | Deploy referenced service first; reference variables resolve only after the referenced service has a stable identity |
| `railway run` doesn't see env var that exists in dashboard | Variable is "sealed" (write-only) — `railway run` cannot read sealed vars by design | Unseal the var (loses sealing benefit) or read from a different mechanism |
| `make deploy-all` succeeds but one service is the old version | Service had a build error; Railway kept the previous deploy | `railway logs --build -s <service>` to find the error |
| Image size > Railway's limit | Layered Dockerfile with `node_modules` in the runtime stage | Multi-stage build; copy only `dist/` and prod deps to runtime |

## DO-NOT list

- DO NOT use `RAILWAY_API_TOKEN` (account scope) when a project token works. The Makefile uses `RAILWAY_TOKEN` only.
- DO NOT skip `-s <service>` in multi-service repos. Even single-service repos benefit from being explicit.
- DO NOT generate `railway.toml` for projects without a custom backend (Scenarios A / B / D / F / G are out of scope).
- DO NOT run database migrations in parallel from multiple services. Pick one service to own migrations; the others read the migrated schema.
- DO NOT bind to `localhost:PORT`. Always `0.0.0.0:$PORT` for HTTP services.
- DO NOT forget `healthcheckPath` for HTTP services. Railway will route traffic to a broken app without it.
- DO NOT use `railway run` with sealed variables — sealed vars are write-only and `railway run` cannot read them.
- DO NOT set `output: 'standalone'` in `next.config.js` for Vercel-hosted Next.js. Standalone is for Railway/Docker only. For Railway: `next build` then `node .next/standalone/server.js` as the start command (the standalone build copies node_modules into `.next/standalone/`, drastically reducing image size).
- DO NOT cross-volume `mv` in any deploy step (not relevant for Railway, but worth noting for any Make recipe that touches files in CI).
- DO NOT auto-create or auto-rotate Railway tokens. Always user-pasted, always at user's request (see `ci-cd-workflow.md`).
