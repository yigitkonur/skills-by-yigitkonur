# CI/CD workflow — push-to-main → GitHub Actions auto-deploy

The optional final step. After the Makefile is generated and AGENTS.md is normalized, the skill asks once: "Wire CI/CD for push-to-main → auto-deploy via GitHub Actions? (y/n)". If yes, the skill generates `.github/workflows/deploy.yml`, prompts for ONLY the tokens the scenario needs, pushes them to GitHub via `gh secret set`, and commits the workflow. The user never opens the GitHub UI; the agent never writes a token to disk except as a `gh secret set` payload.

This step is opt-in. Decline = skill ends cleanly with `.github/workflows/` untouched. There's no half-baked YAML from a partial wiring.

## The trigger flow

```
1. Detect repo on GitHub:        gh repo view --json nameWithOwner -q .nameWithOwner
                                  └─ if not a GitHub repo: print manual-setup hint, skip CI/CD entirely
2. Confirm gh authenticated:     gh auth status
                                  └─ if unauthenticated: ask user to `gh auth login` first; halt CI/CD
3. Ask the user:                 "Wire CI/CD for push-to-main → auto-deploy via GitHub Actions? (y/n)"
                                  └─ default n; only proceed on explicit y
4. Determine providers:          (Vercel | Railway | Supabase | combination, per scenario)
5. Generate workflow YAML:       .github/workflows/deploy.yml — only blocks for in-scope providers
6. Prompt for tokens:            one prompt per provider, using exact wording below
7. Push secrets via gh:          gh secret set <NAME> --body "$pasted"
8. Verify secrets present:       gh secret list (prints names only, never values)
9. Commit + push:                git add .github/workflows/deploy.yml && git commit && git push origin main
10. Print closing banner:        wired-secrets list, rotation hint
```

Each step is a hard gate. Step 7 fails closed: if any `gh secret set` errors, halt before step 8 — don't push a YAML that references secrets that don't exist (Actions runs would fail at runtime with confusing "secret X not found" errors, far from the wiring step where the user could fix it).

## Provider scope by scenario

| Scenario | Workflow includes |
|---|---|
| A — Frontend-only | Vercel block |
| B — MCP server | (No deploy — skip CI/CD entirely; ask the user; the skill prints "Scenario B has no remote deploy target — skipping CI/CD wiring.") |
| C — Frontend + backend | Vercel + Railway blocks |
| D — Frontend + Supabase | Vercel + Supabase blocks |
| E — Multi-service Railway | Railway block, repeated per service (one `railway up --ci -s <name>` per service) |
| F — Build-artifact | (No deploy — skip CI/CD; same message as Scenario B) |
| G — MacBook ship | (No remote-deploy target — skip CI/CD; the ship pipeline is local-build-only by design) |

The skill never generates a workflow with a Railway block for Scenario A or a Vercel block for Scenario E. The YAML is templated to match the scenario's actual deploy targets — anything else is dead code.

## The full workflow YAML template

The generated file at `.github/workflows/deploy.yml`. Only the blocks for in-scope providers are kept; commented dividers (`# ─ Vercel ─`) are placeholders showing where to splice. The skill renders the final YAML with provider-specific blocks present and others omitted.

```yaml
# Generated by init-makefiles. Edit cautiously.
name: Deploy
on:
  push:
    branches: [main]
  workflow_dispatch:
concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: false
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      # Provider-specific steps follow:
      # ─ Vercel ─
      - run: npm i -g vercel
      - run: vercel deploy --prod --yes --token=$VERCEL_TOKEN
        env:
          VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
      # ─ Railway ─ (one block per service in scenario E)
      - run: bash <(curl -fsSL railway.com/install.sh) -y
      - run: railway up --ci -s api
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
      # ─ Supabase ─
      - run: npm i -g supabase
      - run: supabase link --project-ref $SUPABASE_PROJECT_REF
      - run: supabase db push --linked
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_DB_PASSWORD: ${{ secrets.SUPABASE_DB_PASSWORD }}
          SUPABASE_PROJECT_REF: ${{ secrets.SUPABASE_PROJECT_REF }}
```

Line-by-line rationale:

| Line | Rationale |
|---|---|
| `on: push: branches: [main]` | Only `main` triggers prod deploys. Other branches and PRs are explicitly excluded — no accidental "preview" prod deploys. |
| `workflow_dispatch:` | Manual run from the Actions UI is preserved — useful for re-deploys after secret rotation, even when the code didn't change. |
| `concurrency: group: deploy-${{ github.ref }}` | One in-flight deploy per branch. Two pushes to main in rapid succession queue rather than race. |
| `cancel-in-progress: false` | The earlier deploy completes; the later one queues. Critical for Railway — cancelling mid-deploy can leave services in a half-replaced state. (See "concurrency rationale" below.) |
| `runs-on: ubuntu-latest` | Cheapest, fastest, broadest tool compatibility. macOS and Windows runners cost more and aren't needed for Vercel/Railway/Supabase deploys. |
| `actions/checkout@v4` | Pinned major version. v4 is current as of May 2026; v3 still works but v4 is the published default. |
| `actions/setup-node@v4 with: { node-version: '20' }` | Node 20 is LTS through April 2026; aligns with Vercel's default Node version and most Railway Railpack-built Node services. |
| `npm i -g vercel` (in the Vercel block) | The Vercel CLI is the canonical deploy tool. `npm i -g` per-job is faster than caching for a small CLI; the install takes ~5s. |
| `vercel deploy --prod --yes --token=$VERCEL_TOKEN` | `--prod` for production deploys (vs preview), `--yes` to skip the "set up and deploy?" prompt, `--token=$VERCEL_TOKEN` for non-interactive auth. |
| `bash <(curl -fsSL railway.com/install.sh) -y` (Railway block) | Official Railway install. `-y` is the unattended flag for the install script. `-fsSL` are the standard curl flags for "fail silently, show errors, follow redirects, location-aware" — you want all four for an installer-curl. |
| `railway up --ci -s <service>` | `--ci` is non-interactive deploy mode (no progress bar, exits on completion); `-s` selects the service. For Scenario E, this step repeats per service. |
| `npm i -g supabase` (Supabase block) | Supabase CLI is npm-distributed. |
| `supabase link --project-ref $SUPABASE_PROJECT_REF` then `supabase db push --linked` | Two-step. The first establishes the project context via the access token (which auto-authenticates from `$SUPABASE_ACCESS_TOKEN` env); the second applies migrations. |

For Scenario E with N services, the Railway block looks like:

```yaml
      # ─ Railway ─ (Scenario E — one run-step per service)
      - run: bash <(curl -fsSL railway.com/install.sh) -y
      - run: railway up --ci -s api
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
      - run: railway up --ci -s worker
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
      - run: railway up --ci -s scheduler
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
```

The skill detects the service list via `railway service list` (if locally linked) or asks the user for the service names if not. Each service deploys sequentially; parallelism would speed up the action but complicates failure handling — sequential keeps the deploy log readable.

## Token-prompt sequence

The skill stops the workflow and prompts once. The exact wording (read it aloud — every word is deliberate):

```
init-makefiles needs the following GitHub Actions secrets to enable
push-to-main deploys. Paste each value when prompted. To skip a
provider, hit enter.

VERCEL_TOKEN  (https://vercel.com/account/tokens):
RAILWAY_TOKEN  (Project → Settings → Tokens — PROJECT-scoped, not account):
SUPABASE_ACCESS_TOKEN  (https://supabase.com/dashboard/account/tokens):
SUPABASE_DB_PASSWORD  (Project → Settings → Database):
SUPABASE_PROJECT_REF  (Project → Settings → General → Reference ID):
```

Wording rules:

- **One prompt block, not five separate questions.** The user pastes each on its own line. Lower-friction than five separate questions.
- **Each line gives the URL or location of the value.** The user doesn't have to context-switch to find where each token lives.
- **`PROJECT-scoped, not account` for `RAILWAY_TOKEN`.** This is the most common token-confusion mistake — `RAILWAY_API_TOKEN` (account scope) and `RAILWAY_TOKEN` (project scope) are different things and only the project-scoped one works for `railway up --ci -s`.
- **`hit enter` to skip.** A blank value triggers a skip — the agent doesn't try to set an empty secret and doesn't include the corresponding YAML block. This handles the case where, e.g., the scenario is "Frontend + Supabase" but the user only wants Vercel wired and not Supabase.
- **No re-confirmation prompt.** The user pastes the value once; the agent uses it once; the value is then handed to `gh secret set` and dropped. No "are you sure?" — fewer steps, same security.

The skill collects only the tokens for in-scope providers. For Scenario A, only `VERCEL_TOKEN` is asked. For Scenario E, only `RAILWAY_TOKEN`. For Scenario D, `VERCEL_TOKEN` plus the three Supabase values.

## The `gh secret set` invocation

For each non-empty pasted value:

```bash
gh secret set VERCEL_TOKEN --body "$pasted"
```

Why `--body` and not stdin: `--body` accepts the value as a CLI argument, transmitted through `gh` over HTTPS without ever being written to disk by `gh` itself. Stdin (`echo "$value" | gh secret set VERCEL_TOKEN`) is also fine but the agent's tooling logs may capture stdin in pipe-debugging output (not always, but possible).

For paranoid users (paid security audits, etc.), the `--body-file` form lets the agent write the value to a tempfile, set the secret, and delete the tempfile in three lines:

```bash
tmp=$(mktemp); printf '%s' "$pasted" > "$tmp"; gh secret set VERCEL_TOKEN --body-file "$tmp"; rm -f "$tmp"
```

Functionally equivalent. The skill defaults to `--body` since it's a single line and `gh` handles the value safely.

The agent NEVER writes the value to a permanent file. NEVER `echo`s it back to the user (not even to confirm — the next step's `gh secret list` confirms by name only).

## Verification — `gh secret list`

After all secrets are set:

```bash
gh secret list
# Output (names only — never values):
# NAME                       UPDATED
# RAILWAY_TOKEN              less than a minute ago
# SUPABASE_ACCESS_TOKEN      less than a minute ago
# SUPABASE_DB_PASSWORD       less than a minute ago
# SUPABASE_PROJECT_REF       less than a minute ago
# VERCEL_TOKEN               less than a minute ago
```

The skill parses the output to confirm the expected secret names appear. If any expected secret is missing, the skill prints which one and halts before pushing the workflow.

GitHub Actions secrets are encrypted at rest with libsodium sealed boxes; the user cannot read them back via `gh secret view` (no such command exists for secret values — only metadata). The agent confirms by name, never by value.

## Commit and push

```bash
git add .github/workflows/deploy.yml
git commit -m "ci: wire push-to-main deploy via init-makefiles"
git push origin main
```

The commit message follows the skills-repo conventional-commits convention (`ci:` type, scope omitted since CI is the single concern). The skill never commits other unrelated work — `git add .github/workflows/deploy.yml` is the precise, narrow stage.

If `git push` fails (auth issue, branch protection, etc.), the skill stops and surfaces the failure. The workflow file is committed locally; the user re-runs `git push` after fixing.

After successful wiring:

The list is the actual set of secrets pushed (skipped tokens — those the user blanked — are NOT listed). The rotation hint shows the same `gh secret set --body` form, so rotating is a one-line follow-up. The UI link is for users who prefer the dashboard.

The banner does NOT echo any token values, even partially. No "ends in `...xy`" hint. No first-N-chars prefix. Tokens never round-trip from the agent's memory back to the user's terminal.

## Concurrency rationale — why `cancel-in-progress: false`

The default in `concurrency:` is `cancel-in-progress: false`, but the skill writes it explicitly to make the choice visible.

| Setting | Behavior | Trade-off |
|---|---|---|
| `cancel-in-progress: false` (default) | New deploys queue; in-flight ones complete. | Pro: no half-deployed Railway services from a cancelled-mid-replace. Con: a 10-minute deploy followed by a quick fix waits 10+ min. |
| `cancel-in-progress: true` | New deploys cancel in-flight ones. | Pro: latest commit deploys faster. Con: can leave Railway services in "deploying" state if the cancel hits during the replace step. |

The skill defaults to `false` for safety. For projects with high merge frequency to main where time-to-deploy matters more than mid-flight protection, the user can flip to `true` after the fact:

```yaml
concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: true   # accept the half-deploy risk for faster TTD
```

This is a per-project judgment call; the skill picks the safe default and documents the alternative.

## Native vs Action-driven — disable provider native integrations

Railway and Vercel both have native GitHub integrations that auto-deploy on push. **If both the native integration and this Action are enabled, every push to main triggers TWO deploys.** This is not just wasted compute — for Railway, simultaneous deploys can race the rolling-replace step.

Recommendation in the closing banner (printed only if the scenario uses Railway):

```
Heads-up: Railway's native GitHub integration also deploys on push.
Disable it in Railway Dashboard → Project → Settings → Service →
Source so this Action is the single source of truth.

Vercel's native integration also creates parallel preview deploys for
PRs. That's fine — it's preview, not prod, and runs on a different
trigger. Leave Vercel's integration on.
```

For Vercel, leaving the native integration on is *helpful* — preview deploys for PRs are independent of the Action's main-only prod deploy. They don't conflict.

## Security notes — what's safe and what's not

- **Token transmission:** `gh secret set` sends the value over HTTPS to GitHub's API. The TLS handshake protects the value in flight; GitHub encrypts it at rest with libsodium sealed boxes (per `docs.github.com/en/actions/security-guides/encrypted-secrets`).
- **Token storage on the agent's host:** the agent holds the value briefly in memory (the variable holding the pasted text). Once `gh secret set` returns, the variable is unset (or goes out of scope). No file on disk.
- **Token visibility in CI logs:** GitHub Actions automatically masks any `${{ secrets.X }}` reference in logs (replaces with `***`). The Vercel/Railway/Supabase CLIs do not, by default, print their tokens — but if a user adds `set -x` or `echo $VERCEL_TOKEN` to a custom step, GitHub's mask catches it. (Don't rely on this; the Action template never echoes a secret.)
- **Token rotation:** users rotate by re-running `gh secret set <NAME> --body "<new value>"` (the closing banner reminds them). The next workflow run picks up the new value automatically — no Action edit needed.
- **Token revocation:** users revoke by deleting the GitHub secret (`gh secret delete <NAME>`) AND revoking on the provider side (Vercel/Railway/Supabase dashboard). The workflow run that needs the deleted secret will fail visibly with a "secret not found" error.
- **Agent memory:** the agent never writes the pasted value to a permanent state file. The CLI session ends; the value is gone from the agent's process.

## Failure modes

| Failure | Symptom | Fix |
|---|---|---|
| `gh auth status` returns "not authenticated" | Step 2 halts | Tell user to run `gh auth login` interactively, then re-run the skill (start at step 2). |
| `gh repo view` fails with "no GitHub remote" | Step 1 halts | Print: "manual setup required for non-GitHub repos." Skill skips CI/CD; Makefile + AGENTS.md are still complete. |
| `gh secret set` fails (rate-limited, permissions, token-too-large) | Step 7 halts mid-way | Print the `gh` error verbatim. Halt the wiring. Do NOT push the workflow YAML. The user fixes (auth, scope, token-source) and re-runs. |
| `gh secret list` shows the secrets but `git push` fails (branch protection) | Step 9 halts | The workflow YAML is committed locally; tell user to push manually after handling branch protection (typically: PR + merge instead of direct push). |
| Action runs but deploy fails | Symptom appears in the Actions UI, not at wiring time | The skill is done; this is a deploy-failure issue. Direct user to the failing step in the Actions UI; common causes are wrong token scope (account-scoped Railway token instead of project-scoped) and missing dashboard config (Vercel project not linked, Railway service name wrong). |
| User accidentally pasted the wrong token | The Action runs but provider auth fails | User runs `gh secret set <NAME> --body "<correct value>"` to overwrite. The closing banner's rotation hint covers this. |

## DO-NOT list

- **DO NOT auto-rotate or auto-create provider tokens.** Always user-pasted, always at user request. The agent has no business creating tokens on the user's behalf — the user owns the provider account, the user owns token lifecycle.
- **DO NOT proceed with secret setting if `gh auth status` is unauthenticated.** Step 2 halts. The user runs `gh auth login` first; the skill waits.
- **DO NOT push `.github/workflows/deploy.yml` until `gh secret list` confirms all required secrets are set.** A workflow that references missing secrets fails at runtime with confusing errors; halt at wiring time when the failure is debuggable.
- **DO NOT generate workflow YAML for providers the scenario doesn't use.** Scenario A's workflow has only the Vercel block; Scenario E's has only Railway. No dead provider blocks.
- **DO NOT echo secrets in the closing banner, in logs, or anywhere user-visible.** The banner prints names only. `gh secret list` prints names only. The agent's chat output prints names only.
- **DO NOT write tokens to disk.** `gh secret set --body` transmits the value without local persistence. If `--body-file` is used, `rm -f` the tempfile immediately after `gh secret set` returns.
- **DO NOT `git push --force` from this workflow generation.** Plain `git push origin main` only. Branch protection might require a PR — handle that as a failure mode and surface to the user.
- **DO NOT add `pull_request` triggers to `deploy.yml`.** PRs deploy to *previews* (Vercel handles via its native integration), not to prod. The workflow is for `push: branches: [main]` only.
- **DO NOT skip the `gh repo view` preflight.** A non-GitHub repo (Bitbucket, GitLab, self-hosted) can't accept `gh secret set`. Detect early and skip cleanly.
- **DO NOT generate the workflow file if the user declines the `(y/n)` question.** Decline = `.github/workflows/` untouched. No half-baked YAML, no orphaned secrets.
- **DO NOT include unused `actions/setup-*` steps.** If the scenario's deploy doesn't use Node, drop `setup-node`. Lean YAML.
