# Supabase — external-CLI workflow for Scenario D

This file is the canonical source for the Supabase target set in any Makefile generated by this skill. It applies to **Scenario D** (frontend + Supabase). The default model: an **external Supabase project** (production, hosted at supabase.com), linked via `supabase link --project-ref <ref>`. Local stack (`supabase start`) is **OFF by default** and gated behind `LOCAL=1` — local-stack development is the exception, not the rule.

All facts below verified May 2026 against `https://supabase.com/docs`.

## What "external by default" means

The Supabase CLI supports two modes:

| Mode | When | Targets the agent generates |
|---|---|---|
| **External (linked)** — DEFAULT | Project lives at supabase.com; CLI runs `--linked` against the remote DB | All `supabase-*` targets in this file. The `--linked` flag is implicit. |
| **Local** — opt-in | `supabase start` runs the full Postgres + Studio + GoTrue stack on this machine | `supabase-local-up`, `supabase-local-down` only. Other targets refuse if not linked. |

The agent picks external because:

1. The user is shipping frontend code that talks to a real Supabase project. Local stack diverges from prod (different default extensions, different auth provider config) and the divergence is silent.
2. Migrations land at the same URL the production app reads — no "works locally, breaks in prod" gap.
3. Branching (Supabase's preview environment feature) operates on the linked project, not the local stack.

If the user genuinely needs the local stack (offline dev, schema sandbox, snapshot testing), they opt in with `LOCAL=1 make supabase-local-up`. The rest of the targets (`supabase-migrate-apply`, `supabase-functions`, `supabase-secrets-*`) continue to operate on the linked remote project.

Reference: `https://supabase.com/docs/reference/cli/supabase-link` (2026-05-08).

## Target inventory

The 11 `supabase-*` targets the skill scaffolds. Each works with zero arguments (sole exceptions: `supabase-migrate-new name=<n>` and `supabase-function name=<n>` — naming a new artifact has no good default).

| Target | Reads | Writes | Idempotent | CONFIRM=1? |
|---|---|---|---|---|
| `supabase-link` | `SUPABASE_PROJECT_REF`, `SUPABASE_ACCESS_TOKEN` | `supabase/.temp/project-ref` (CLI's local link state) | yes | no |
| `supabase-status` | linked state OR local stack | none | yes | no |
| `supabase-migrate-new name=<n>` | none | `supabase/migrations/<timestamp>_<name>.sql` | yes (new file) | no |
| `supabase-migrate-plan` | linked DB | none (read-only) | yes | no |
| `supabase-migrate-apply` | linked DB, `CONFIRM=1` | runs migrations on remote DB | yes | YES |
| `supabase-pull` | linked DB | new migration file under `supabase/migrations/` | yes | no |
| `supabase-functions` | `supabase/functions/` | deploys all functions to linked project | yes | no |
| `supabase-function name=<n>` | `supabase/functions/<n>/` | deploys single function | yes | no |
| `supabase-types` | linked DB | `$(SUPABASE_TYPES_PATH)` (default `src/db/types.ts`) | yes | no |
| `supabase-secrets-list` | linked project | none (lists names; values masked) | yes | no |
| `supabase-secrets-push` | `supabase/.env`, `CONFIRM=1` | overwrites server-side secrets | yes | YES |

## Default knobs

```makefile
SUPABASE_PROJECT_REF ?=                                # required for `supabase-link`; agent prompts on first run if blank
SUPABASE_TYPES_PATH  ?= src/db/types.ts                # where `supabase-types` writes
SUPABASE_FUNCTIONS_DIR ?= supabase/functions           # where `supabase-functions` deploys from
SUPABASE_SECRETS_FILE ?= supabase/.env                 # what `supabase-secrets-push` reads
```

`SUPABASE_PROJECT_REF` has no default — it's the unique reference ID of the linked project (Project → Settings → General → Reference ID). The Makefile refuses to run any `supabase-*` target if it's blank AND the project isn't already linked. `SUPABASE_ACCESS_TOKEN` is read from env if set; otherwise the CLI prompts via `supabase login`.

## `supabase-link` — connect to the remote project

Idempotent. Re-running on an already-linked project no-ops.

```makefile
supabase-link:
	@command -v supabase >/dev/null 2>&1 || { \
	  printf "$(R)supabase CLI not found$(Z)  $(D)brew install supabase/tap/supabase$(Z)\n"; exit 1; }
	@if [ -z "$${SUPABASE_ACCESS_TOKEN:-}" ] && ! supabase login --token "$${SUPABASE_ACCESS_TOKEN:-}" >/dev/null 2>&1; then \
	  printf "$(D)→ supabase login (one-time)$(Z)\n"; \
	  supabase login; \
	fi
	@if [ -z "$(SUPABASE_PROJECT_REF)" ]; then \
	  printf "$(R)SUPABASE_PROJECT_REF not set$(Z)\n"; \
	  printf "$(D)Find it at: Project → Settings → General → Reference ID$(Z)\n"; \
	  printf "$(D)Then: SUPABASE_PROJECT_REF=<ref> make supabase-link$(Z)\n"; \
	  exit 1; \
	fi
	@supabase link --project-ref $(SUPABASE_PROJECT_REF)
	@printf "$(G)✓$(Z) linked to $(SUPABASE_PROJECT_REF)\n"
```

Reference: `https://supabase.com/docs/reference/cli/supabase-link` (2026-05-08).

## `_check-supabase-linked` — gate every other target

Most targets need a linked project. This helper exits cleanly if the link state is missing.

```makefile
_check-supabase-linked:
	@if [ ! -f supabase/.temp/project-ref ] && [ -z "$(SUPABASE_PROJECT_REF)" ]; then \
	  printf "$(R)supabase project not linked$(Z)\n"; \
	  printf "$(D)Run: make supabase-link  (or set SUPABASE_PROJECT_REF and re-run)$(Z)\n"; \
	  exit 1; \
	fi
```

Every linked-only target prerequisites this:

```makefile
supabase-migrate-plan: _check-supabase-linked
	@supabase db push --linked --dry-run
```

## `supabase-status` — what's linked / what's running

Two modes:

```makefile
supabase-status:
	@if [ "$${LOCAL:-0}" = "1" ]; then \
	  supabase status; \
	else \
	  supabase projects list 2>/dev/null | head -20; \
	fi
```

Default (no `LOCAL=1`) prints the user's accessible projects with the linked one marked. With `LOCAL=1`, prints the local stack's port mappings. Refuses local-stack inspection unless `LOCAL=1` so a stray `make supabase-status` doesn't accidentally start querying a local Postgres that isn't running.

## `supabase-migrate-new` — create a new migration

The lone exception to the no-arg rule. Migration filenames have no good default.

```makefile
supabase-migrate-new:
	@if [ -z "$(name)" ]; then \
	  printf "$(R)usage: make supabase-migrate-new name=<descriptive_name>$(Z)\n"; exit 1; \
	fi
	@supabase migration new $(name)
	@printf "$(G)✓$(Z) new migration: supabase/migrations/<timestamp>_$(name).sql\n"
```

The user runs:

```bash
make supabase-migrate-new name=add_users_table
```

The CLI writes `supabase/migrations/20260508120000_add_users_table.sql`. The user edits the SQL, then plans + applies (next two targets).

## `supabase-migrate-plan` — dry-run against the linked DB (READ-ONLY)

```makefile
supabase-migrate-plan: _check-supabase-linked
	@printf "$(D)→ dry-run: supabase db push --linked --dry-run$(Z)\n"
	@supabase db push --linked --dry-run
```

`--dry-run` calculates the diff and prints the SQL that *would* run, but never applies it. Use as a pre-flight before the apply step. Read-only; no `CONFIRM=1` gate.

Reference: `https://supabase.com/docs/reference/cli/supabase-db-push` (2026-05-08).

## `supabase-migrate-apply` — apply migrations to the linked DB (REQUIRES CONFIRM=1)

The destructive target. Refuses to run without `CONFIRM=1`.

```makefile
supabase-migrate-apply: _check-supabase-linked
	@if [ "$${CONFIRM:-0}" != "1" ]; then \
	  printf "$(R)refusing to apply migrations without explicit confirmation$(Z)\n"; \
	  printf "$(D)→ first review the diff:  make supabase-migrate-plan$(Z)\n"; \
	  printf "$(D)→ then apply:             CONFIRM=1 make supabase-migrate-apply$(Z)\n"; \
	  exit 1; \
	fi
	@printf "$(Y)→ applying migrations to $(SUPABASE_PROJECT_REF)$(Z)\n"
	@supabase db push --linked
	@printf "$(G)✓$(Z) migrations applied\n"
```

The gate is intentionally simple: one env var, one explicit value. The user types `CONFIRM=1 make supabase-migrate-apply` once they've reviewed the plan. No second nag, no interactive prompt that automation can choke on.

`db push --linked` is non-destructive in the sense that it only adds new migrations on top of the remote history. But the migrations themselves can DROP COLUMN, ALTER TYPE, TRUNCATE — destructive SQL applies silently if the user wrote it. Always run `--dry-run` first.

## `supabase-pull` — fetch remote schema as a new migration

Read-only against the remote DB. Writes a migration file locally that captures the drift.

```makefile
supabase-pull: _check-supabase-linked
	@printf "$(D)→ pulling remote schema as a new migration$(Z)\n"
	@supabase db pull --linked
	@printf "$(G)✓$(Z) pulled — new migration in supabase/migrations/\n"
```

Use case: someone edited the schema via the Dashboard SQL editor (bypassing migration history). `supabase-pull` regenerates the migration file from the remote state so the local history catches up. Always commit the resulting migration immediately.

## `supabase-functions` and `supabase-function name=<n>` — Edge Functions deploy

Deploy all functions:

```makefile
supabase-functions: _check-supabase-linked
	@printf "$(D)→ deploying all functions in $(SUPABASE_FUNCTIONS_DIR)/$(Z)\n"
	@supabase functions deploy --linked
	@printf "$(G)✓$(Z) functions deployed\n"
```

Deploy one function (the `name=` exception):

```makefile
supabase-function: _check-supabase-linked
	@if [ -z "$(name)" ]; then \
	  printf "$(R)usage: make supabase-function name=<function_name>$(Z)\n"; exit 1; \
	fi
	@supabase functions deploy $(name) --linked
	@printf "$(G)✓$(Z) function $(name) deployed\n"
```

Reference: `https://supabase.com/docs/reference/cli/supabase-functions-deploy` (2026-05-08).

The agent detects functions by `ls $(SUPABASE_FUNCTIONS_DIR)`. If empty, the targets are still generated but a one-line hint is printed during scaffolding: "no functions detected; targets are placeholders — drop functions in `supabase/functions/<name>/index.ts` and re-run".

## `supabase-types` — generate TypeScript types from the schema

```makefile
supabase-types: _check-supabase-linked
	@printf "$(D)→ generating types from linked schema → $(SUPABASE_TYPES_PATH)$(Z)\n"
	@mkdir -p $$(dirname "$(SUPABASE_TYPES_PATH)")
	@supabase gen types typescript --linked > $(SUPABASE_TYPES_PATH)
	@printf "$(G)✓$(Z) types written to $(SUPABASE_TYPES_PATH)\n"
```

The agent picks the path from `SUPABASE_TYPES_PATH` (default `src/db/types.ts`). Typical project layouts:

| Project shape | Recommended path |
|---|---|
| Next.js with `src/` dir | `src/db/types.ts` |
| Next.js without `src/` | `lib/db/types.ts` |
| Vite/React standalone | `src/db/types.ts` |

Run after every `supabase-migrate-apply` so types stay in sync with the schema. Add to CI as a sanity check (the generated file should be commit-clean):

```bash
make supabase-types && git diff --exit-code $(SUPABASE_TYPES_PATH)
```

If the diff is non-empty, someone applied a migration without regenerating types — fix the workflow (or wire `supabase-types` into a pre-commit hook).

Reference: `https://supabase.com/docs/reference/cli/supabase-gen-types-typescript` (2026-05-08).

## `supabase-secrets-list` and `supabase-secrets-push`

List the names of currently-set secrets (values are masked):

```makefile
supabase-secrets-list: _check-supabase-linked
	@supabase secrets list
```

Push secrets from a local `.env` file. Refuses without `CONFIRM=1`:

```makefile
supabase-secrets-push: _check-supabase-linked
	@if [ "$${CONFIRM:-0}" != "1" ]; then \
	  printf "$(R)refusing to overwrite remote secrets without explicit confirmation$(Z)\n"; \
	  printf "$(D)→ review:   cat $(SUPABASE_SECRETS_FILE)$(Z)\n"; \
	  printf "$(D)→ apply:    CONFIRM=1 make supabase-secrets-push$(Z)\n"; \
	  exit 1; \
	fi
	@if [ ! -f "$(SUPABASE_SECRETS_FILE)" ]; then \
	  printf "$(R)$(SUPABASE_SECRETS_FILE) not found$(Z)\n"; \
	  printf "$(D)Create it from a template; make sure it's gitignored.$(Z)\n"; \
	  exit 1; \
	fi
	@supabase secrets set --env-file $(SUPABASE_SECRETS_FILE)
	@printf "$(G)✓$(Z) secrets pushed from $(SUPABASE_SECRETS_FILE)\n"
```

Reference: `https://supabase.com/docs/reference/cli/supabase-secrets` (2026-05-08).

## Local stack (opt-in) — `supabase-local-up` / `supabase-local-down`

Both gated on `LOCAL=1`. Without it, the targets refuse cleanly so a stray `make supabase-local-up` doesn't accidentally pull a 1.5 GB Docker image.

```makefile
supabase-local-up:
	@if [ "$${LOCAL:-0}" != "1" ]; then \
	  printf "$(R)local Supabase stack is opt-in$(Z)\n"; \
	  printf "$(D)This skill defaults to the linked remote project.$(Z)\n"; \
	  printf "$(D)To start the local stack anyway:  LOCAL=1 make supabase-local-up$(Z)\n"; \
	  exit 1; \
	fi
	@command -v docker >/dev/null 2>&1 || { \
	  printf "$(R)docker not found — local stack requires Docker$(Z)\n"; exit 1; }
	@supabase start

supabase-local-down:
	@if [ "$${LOCAL:-0}" != "1" ]; then \
	  printf "$(R)local Supabase stack is opt-in (LOCAL=1 required)$(Z)\n"; \
	  exit 1; \
	fi
	@supabase stop
```

The local stack also needs `supabase/config.toml` at the project root — `supabase init` creates it, the user commits it. The Makefile assumes it exists (Scenario D detection requires `supabase/` dir).

Note: when the local stack is running, *other* targets (migrate, functions, types) still default to `--linked` (the remote). The user must explicitly switch context if they want the local stack to be the target — but the skill doesn't generate a "switch context" target because it's a footgun. Linked is the source of truth.

## Branching — May 2026 model

As of May 2026, Supabase's branching feature defaults to **branching without git** (each preview branch in your code creates its own Supabase instance, automatically):

> "branching without git is now the default" — `https://supabase.com/blog/branching-without-git-is-now-the-default` (2026-05)

What this means for the Makefile:

- **Each branch deploy gets its own Supabase project instance** (preview branches in Vercel auto-link to a fresh Supabase branch via the Vercel-Supabase integration).
- **The Vercel preview environment auto-syncs** the branch's `SUPABASE_URL` and `SUPABASE_ANON_KEY` into the preview build's env. No `make` plumbing required.
- **Production stays separate** — your production code talks to the production Supabase project; preview branches talk to their own.

Implication: **the Makefile only operates on the linked production project**. Don't try to manage branches from `make`. The Vercel-Supabase integration owns branch lifecycle. The Makefile's `SUPABASE_PROJECT_REF` is the production reference; never branch-specific.

If a user needs to inspect a branch's DB or copy its schema, they do so via the Dashboard or via a one-off `supabase link --project-ref <branch-ref>` in a separate working directory — outside the scope of this Makefile.

## Migration safety rules

| Rule | Why |
|---|---|
| Always `--dry-run` first | `db push` is idempotent against new migrations but the migrations themselves can be destructive |
| Never `db reset` against a remote connection string | `db reset` drops ALL tables and re-runs the full migration history. Catastrophic in prod. |
| Never edit production schema via Dashboard SQL | Bypasses migration history; future deploys will skew. If you must, run `supabase-pull` immediately after to capture the drift. |
| One migration per intent | A migration that adds 3 unrelated tables is harder to revert than 3 migrations with one each |
| Test migrations against a clone first | Use Supabase's branching (preview env) or a manual `pg_dump` + `psql` round-trip on staging |
| Never run two `db push --linked` simultaneously | The migration table is the lock; the second call may fail or worse, corrupt the history |

The skill **never** generates a `db reset` target. Reset is for local stack only and is dangerous enough to require manual invocation.

## The `supabase/.env` convention

Local file holding secrets the Edge Functions need at runtime. Pushed to the linked project via `supabase-secrets-push`.

```
supabase/
├── config.toml           ← committed
├── .env                  ← gitignored — secrets
├── .env.template         ← committed — keys-only template (no values)
├── migrations/
│   └── *.sql
└── functions/
    └── <name>/
        └── index.ts
```

Recommended `supabase/.env.template`:

```
# Edge Function runtime secrets
# Copy to supabase/.env and fill in values.
# Never commit supabase/.env.

STRIPE_SECRET_KEY=
RESEND_API_KEY=
INTERNAL_WEBHOOK_SECRET=
```

The Makefile's `supabase-secrets-push` reads `supabase/.env` (the gitignored copy), pushes via `supabase secrets set --env-file`. The template is the spec; the actual values live in 1Password (or the user's preferred secret store) and get pasted into `supabase/.env` per machine.

`supabase/.env` lines are also where the agent suggests adding to `.gitignore` (alongside `.env.local`, `.vercel/`, etc.) — see `env-vars-conventions.md`.

## Common failure modes

| Symptom | Root cause | Fix |
|---|---|---|
| `supabase link` errors with "project not found" | Wrong `SUPABASE_PROJECT_REF` or unauthorized token | Verify ref in Dashboard → Settings → General; verify token in Account → Tokens |
| `db push --linked` errors with "no migrations found" | `supabase/migrations/` empty | Run `make supabase-migrate-new name=<n>` to create one |
| `db push --linked` no-ops | All migrations already in remote history | Run `make supabase-pull` to detect drift; if no drift, you're up to date |
| `gen types typescript` produces empty output | Linked project has no public schema tables | Apply migrations first, then regen |
| `secrets push` overwrites unintended keys | `--env-file` upserts every key in the file (doesn't diff against remote) | Review `supabase/.env` carefully; use `supabase-secrets-list` to see what's there before pushing |
| Edge Function deploy succeeds but throws at runtime | Missing secret (set locally, never pushed) | `make supabase-secrets-list` shows what's set; push missing ones |
| `service_role` key leaked into the frontend bundle | `NEXT_PUBLIC_*` envs are public; never put `SUPABASE_SERVICE_ROLE_KEY` there | Rotate immediately (Dashboard → Settings → API → Reset service role key); fix the leak path; use `SUPABASE_SERVICE_ROLE_KEY` only on server / Edge Functions |
| Production schema diverges from migration history | Someone ran SQL via Dashboard | `make supabase-pull` to recover; commit the new migration; institute a "no Dashboard SQL" rule |

## DO-NOT list

- DO NOT run `db reset` against a remote connection string. Local stack only.
- DO NOT leak the `SUPABASE_SERVICE_ROLE_KEY` to the frontend. It bypasses RLS and grants full DB access. Server / Edge Functions only.
- DO NOT skip `--dry-run` on schema-changing migrations. `make supabase-migrate-plan` first, then `CONFIRM=1 make supabase-migrate-apply`.
- DO NOT manage auth providers / storage policies / realtime config via CLI. Most are NOT supported by `supabase` CLI in 2026 — manage via Dashboard or the Management API. The Makefile does not generate targets for these.
- DO NOT bypass the `CONFIRM=1` gate by aliasing `supabase-migrate-apply` or `supabase-secrets-push` to a no-gate variant. The gate is the safety net.
- DO NOT commit `supabase/.env`. Gitignore it; commit `supabase/.env.template` instead.
- DO NOT push secrets while logged in to the wrong project. Run `make supabase-secrets-list` first to confirm the linked project ref matches what you expect.
- DO NOT auto-link to a new project from the Makefile. Linking is one-time, user-initiated (`make supabase-link`).
- DO NOT edit production schema via Dashboard SQL. Always migration files. If you must, `make supabase-pull` immediately after.
- DO NOT generate Supabase migrations in projects without a `supabase/` dir. Skip the targets entirely (skill detection catches this).
